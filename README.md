# GHGA Service Dashboard

A single page dashboard for the GHGA service repository landscape.

- **Latest release** — version + date (GitHub's pinned "Latest", not newest-by-date)
- **Test-bed** — the version pinned in the integration test-bed compose, marked
  ✓ latest / ▼ behind / ▲ ahead relative to the latest release
- **Latest pre-release** — version + date
- **Open PRs** — count (expand a row to see them)
- **Main vs release** — whether `main` has commits not yet released (▲ N unreleased)

For the **`file-services-backend`** monorepo it also lists each service's
version (from `services/<svc>/pyproject.toml`) at the latest release tag vs. on
`main` vs. the test-bed, so you can see which service drifted.

## Using the dashboard

**Hosted:** open <https://ghga-de.github.io/service-dashboard/>.

The published page is **fully static** — it reads a prebuilt `data.json` and
**never handles a GitHub token in the browser.** The data is generated by a
GitHub Actions workflow that runs `dashboard.py build` with an org-owned,
read-only token kept in Actions secrets (see [Deployment](#deployment-github-pages)).
The header shows when the data was last built.

Because the browser holds no token and makes no live API call, the page can't be
tricked into leaking a credential and there is nothing for a visitor to abuse.
The trade-off is that the data is only as fresh as the last build.

### Refreshing the data

The workflow rebuilds on every push to `main` and on demand — there is no
schedule. To force a refresh, use the **re-run to update** link in the page
header, or open **Actions → Deploy to GitHub Pages → Run workflow** (needs repo
write access).

## Deployment (GitHub Pages)

The site is deployed by [`.github/workflows/pages.yml`](.github/workflows/pages.yml):
it checks out the repo, runs `python3 dashboard.py build data.json` to generate
the payload, and publishes everything to GitHub Pages. The generated `data.json`
is a build artifact and is not committed.

**One-time setup:**

1. **Enable Pages:** repo **Settings → Pages → Source: "GitHub Actions"**.
2. **Add the token secret.** Create a repository secret named **`DASHBOARD_TOKEN`**
   holding a **fine-grained, read-only** GitHub token that can read the tracked
   `ghga-de` repositories (including the private `archive-test-bed`):
   - Permissions: **Metadata: read**, **Contents: read**, **Pull requests: read**.
   - Scope it to the `ghga-de` repositories and give it a short expiry.
   - A classic PAT with `repo` works too but is over-privileged — prefer the
     fine-grained token, or a GitHub App installation token.

The token lives only in the Actions environment and is never shipped to the
browser. Without `DASHBOARD_TOKEN` the build still runs but hits GitHub's 60/hour
unauthenticated limit, so most rows will show errors.

> **Note:** the published `data.json` is public. It includes the test-bed service
> **names and versions** read from the private `archive-test-bed` repo — nothing
> else from that repo is exposed.

## Local backend (optional)

`dashboard.py` provides a Python backend works as a local server if you prefer to run it locally while keeping the token out of the browser entirely:

```bash
export GITHUB_TOKEN=ghp_xxxxxxxx
python3 dashboard.py                  # stdlib only — no pip install
```

Then open **http://127.0.0.1:8888**. The page auto-detects the backend (via
`/healthz`) and uses its `/api/data` endpoint; the token stays in the Python
process. The same `index.html` powers both modes.

### Backend options (environment variables)

| Var | Default | Meaning |
|-----|---------|---------|
| `GITHUB_TOKEN` / `GH_TOKEN` | – | API token |
| `PORT` | `8888` | Port to serve on |
| `HOST` | `127.0.0.1` | Bind address |
| `CACHE_TTL` | `60` | Seconds to cache results (use **Refresh** to force) |
| `MONOREPO_DETAIL` | `1` | Set `0` to skip per-service version lookups |
| `TESTBED_REPO` | `archive-test-bed` | Repo holding the integration test-bed compose |
| `TESTBED_REF` | `main` | Branch/ref to read the compose from |
| `TESTBED_COMPOSE_PATH` | `.devcontainer/docker-compose.yml` | Path to the compose file |
| `HTTP_RETRIES` | `3` | Attempts per request on transient errors |
| `REQUEST_SPACING` | `0.05` | Min seconds between outgoing request starts |
| `LOG_LEVEL` | `INFO` | Backend log verbosity (`DEBUG`/`INFO`/`WARNING`) |

Note: the repository list lives only in `dashboard.py`
(`STANDALONE_REPOS` / `FSB_SERVICES`); the hosted page renders whatever
`data.json` the build produced, so there is no second list to keep in sync.

## Features

### Export

The **Export** button (top right) opens a copy/download panel with a plain-text
list of every service's latest release (the monorepo is expanded into its
services; repos without a release are omitted). Three formats: `image:tag`
(default, e.g. `ghga/auth-service:10.0.0`), `name: version`, and aligned
`name  version`.

### Test-bed column

The version each service runs in the inter-service integration test bed is read
once per refresh from the **`archive-test-bed`** repo's
`.devcontainer/docker-compose.yml` (`main`). Every `ghga/<name>:<version>` image
is matched to its repo (or monorepo service) and compared to the latest release,
so **▼ behind** flags a service the test-bed hasn't been bumped to yet. Repeated
instances (rest/consumer/init) collapse to one unique version.

**Branch override (local backend only).** When running `dashboard.py`, the
**Test-bed branch** box (top right) reads the compose from any branch on demand;
only the Test-bed column re-computes (one extra API call, reusing the cached
release data). The hosted static page always shows the `main` branch's test-bed
versions and hides this control.
